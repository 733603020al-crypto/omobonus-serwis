# CLAUDE.md

## Правила работы

Отвечай на русском. Работай экономно по токенам.

Работай профессионально и критично. Если данных недостаточно — скажи об этом или задай короткий уточняющий вопрос только тогда, когда без этого невозможно безопасно выполнить задачу. Не делай вид, что уверен, если это не так.

По умолчанию работай автономно. Не задавай промежуточные вопросы, если действие безопасно, относится к текущей задаче и не входит в список критичных исключений.

## Автономный режим по умолчанию

Claude всегда должен работать автономно по умолчанию.

Для этого не нужно, чтобы пользователь каждый раз писал:

* `работай автономно`;
* `не спрашивай`;
* `сделай до конца`;
* `без промежуточных вопросов`;
* `commit/push разрешён`.

Эти правила действуют для всех задач, если пользователь прямо не написал обратное.

Claude должен сам:

* определить минимальный набор релевантных файлов;
* прочитать нужные файлы;
* выбрать безопасный технический вариант;
* выполнить задачу;
* исправить локальные ошибки, вызванные текущей правкой;
* запустить проверки;
* проверить `git status --short`;
* при необходимости сделать commit/push в `test`;
* дать короткий финальный отчёт.

Claude не должен задавать промежуточные вопросы вроде:

* можно ли читать релевантные файлы;
* можно ли открыть больше 3 файлов, если они реально нужны для задачи;
* продолжать ли после read-only проверки;
* какой безопасный технический вариант выбрать;
* можно ли запускать `rg`, `find`, `ls`, `node -e`;
* можно ли запускать `npx tsc --noEmit`;
* можно ли запускать `npm run build`;
* можно ли проверять dev routes;
* можно ли исправить очевидную TypeScript/build ошибку, вызванную текущей правкой;
* можно ли вынести повторяющийся JSX в общий template;
* можно ли вынести видимые тексты в i18n/data;
* можно ли сделать общий компонент языковым;
* можно ли синхронизировать существующую языковую версию с польской.

Если есть несколько безопасных вариантов, Claude должен выбрать вариант, который:

1. меньше меняет проект;
2. лучше сохраняет текущий дизайн;
3. лучше поддерживает общий skeleton;
4. не меняет SEO-смысл;
5. не ломает PL как source of truth;
6. проще расширяется на будущие языки;
7. не затрагивает цены, контакты, формы, юридический смысл и `master`.

Claude не должен спрашивать “какой вариант выбрать”, если один из вариантов явно соответствует этим правилам.

### Commit / push по умолчанию

Если Claude изменил файлы проекта и задача не запрещает commit/push, Claude может самостоятельно сделать commit и push в `test`, если выполнены все условия:

* текущая ветка — `test`;
* `git status --short` показывает только ожидаемые файлы текущей задачи;
* нет неожиданных изменений;
* нет изменений в `.env*`;
* нет изменений в `master`;
* не менялись цены, контакты, отправка формы и юридический смысл текста без прямого задания;
* `npx tsc --noEmit` прошёл;
* `npm run build` прошёл.

Commit message Claude выбирает сам, коротко и по смыслу.

Push разрешён только так:

`git push origin test`

Запрещено пушить в `master`.

Если пользователь прямо написал `не commit`, `не push`, `только аудит`, `ничего не менять`, `read-only` или аналогичное ограничение — Claude обязан выполнить это ограничение.

### Git add

Claude не должен использовать `git add -A` вслепую.

`git add -A` разрешён только если Claude сначала проверил `git status --short` и в статусе есть только ожидаемые файлы текущей задачи.

Если изменён один или несколько конкретных файлов, предпочтительно использовать точечный `git add <path>`.

### Когда нужно остановиться и спросить

Claude должен остановиться и спросить пользователя только в критичных случаях:

* текущая ветка не `test`;
* есть неожиданные незакоммиченные изменения;
* нужно трогать `master`;
* нужно читать или менять `.env*`;
* нужно установить, удалить или обновить npm-пакеты;
* нужно изменить цены;
* нужно изменить контакты;
* нужно изменить юридический смысл текста;
* нужно изменить отправку формы или backend API;
* нужно изменить URL, slug, canonical, hreflang, sitemap или robots.txt, если это не было явно частью задачи;
* нужно удалить существующий видимый блок сайта;
* нужно удалить файл, использование которого не проверено через imports / `rg` / git;
* build/typecheck не проходит, и причина не относится очевидно к текущей правке;
* исправление требует большого refactor за пределами текущей задачи;
* изменение выходит за рамки текущего задания.

Если файл или блок выглядит подозрительным, но нет доказательства, что его можно безопасно удалить, Claude не должен спрашивать сразу. Он должен сначала проверить imports, `rg`, git-tracking и usage. Спрашивать нужно только если после проверки остаётся реальный риск.

### Поведение при сомнении

Если сомнение не критичное, Claude должен выбрать минимальный безопасный вариант и продолжить.

Если действие потенциально рискованное, но можно безопасно сделать read-only проверку, Claude должен сначала выполнить read-only проверку, а не спрашивать пользователя.

Если после проверки риск подтвердился, Claude должен остановиться и кратко объяснить причину.

Финальный отчёт должен быть коротким и практическим:

* что изменено;
* какие файлы изменены;
* какие проверки прошли;
* какой commit создан;
* был ли push в `test`;
* что осталось отдельной задачей.

Если работа выполнена успешно, Claude не должен завершать ответ вопросом к пользователю.

## Read-only audit mode

Этот режим применяется, если задача сформулирована как:

* audit;
* проверка лишних файлов;
* поиск мусора;
* поиск неиспользуемых файлов;
* inspection;
* read-only анализ;
* проверка структуры проекта без изменений.

В этом режиме работай автономно до финального отчёта.

Разрешено без дополнительного вопроса выполнять read-only проверки:

* `git status --short`;
* `git branch --show-current`;
* `git log --oneline`;
* `git diff --name-only`;
* `git diff --stat`;
* `ls`;
* `find`;
* `rg`;
* `du`;
* `wc`;
* `head`;
* `tail`;
* `node -e` для анализа файлов;

В этом режиме можно открыть больше 3 релевантных файлов без вопроса, если это нужно для audit-задачи.

Не спрашивай промежуточные подтверждения для read-only команд. Группируй проверки разумно и продолжай до итогового отчёта.

Спрашивай подтверждение только если нужно:

* удалить файл или папку;
* изменить файл;
* сделать commit;
* сделать push;
* установить пакет;
* выполнить `git reset`;
* выполнить `git rebase`;
* выполнить `git clean`;
* выполнить команду, которая может изменить проект;
* читать `.env*`;
* менять код, SEO, URL, canonical, hreflang, юридические тексты, цены, контакты или форму отправки.

Важно:

* Ничего не удаляй без отдельного подтверждения.
* Ничего не меняй без отдельного подтверждения, если задача явно read-only.
* Не делай commit/push, если задача явно read-only.
* Не предлагай удалять `node_modules` как часть очистки проекта.
* Не предлагай удалять `.next` как изменение проекта.
* Не предлагай удалять `public/images` без доказательства, что конкретный файл нигде не используется.
* Не предлагай удалять `CLAUDE.md`.
* Не предлагай удалять файлы правил Claude.

## Безопасность

* Не читай `.env`, `.env.local`, `.env.production`, `.env.development`, ключи, токены, пароли и секреты без моего разрешения.
* Не выводи секреты в ответ.
* Не устанавливай пакеты без подтверждения.
* Не запускай опасные команды: удаление файлов, очистка базы, `git reset`, `git rebase`, `git clean`, деплой, push в `master`, отправка секретов наружу.
* `git push origin test` разрешён после успешной проверки, если задача подразумевает изменение проекта и не запрещает commit/push.
* `master` нельзя трогать вообще.
* Работай только в ветке `test`, если в текущем задании не указано иное.
* Перед изменениями молча проверь текущую ветку.
* Если текущая ветка не `test`, остановись и ничего не меняй.
* Если текущая ветка `test`, продолжай работу автономно и не спрашивай про ветку каждый раз.
* Не читай `.env*` даже в audit-режиме.

### ПРАВИЛО: Автономная проверка результата

Перед началом любой задачи сам определи для себя (не озвучивая мне), что должно остаться неизменным, а что — цель изменения. Базовый принцип: если задача про логику/скорость — внешний вид, расположение, тексты, данные, поведение сайта должны остаться такими же, как были. Не выходи за рамки того, что реально требует задача — не трогай страницы/элементы, не относящиеся к задаче, без необходимости.

После завершения — сам проверь результат (headless Chrome, визуальное сравнение до/после на затронутых страницах и на соседних, если код общий/переиспользуемый). Глубина проверки — по своему усмотрению, пропорционально риску.

Мне в конце пиши только короткий итог в 1-2 строки: что сделано и подтверждение, что проверил и всё в порядке (например: "Ускорил загрузку меню, всё проверено — сайт выглядит и работает как раньше"). Без списков файлов, без технических деталей, если только я сам не спрошу подробнее.

Если в процессе находишь, что задача требует более широких изменений, чем ожидалось изначально, или обнаруживаешь визуальное расхождение, которое не можешь тихо исправить в рамках той же логики — тогда остановись и спроси меня, но только в этом случае.

### ПРАВИЛО: Полная автономность

Всё, кроме мержа в `master` (это правило остаётся в силе), Claude делает сам, без вопросов и без ожидания подтверждения на каждом шаге: чтение файлов, поиск, любые bash-команды (включая headless Chrome тесты), запуск build/tsc, внесение правок в коде, `git commit` и `git push origin test`.

Если есть выбор между несколькими вариантами решения (например, рискованный/безопасный подход) — выбирай сам более безопасный и консервативный вариант, не спрашивая, и отметь это коротко в финальном отчёте, если это существенно.

Работай до полного завершения задачи и присылай короткий финальный итог без списков файлов и технических деталей, если не спросили отдельно.

Это правило действует всегда, для любых задач.

## Изменение сайта

* Не давай длинный план перед каждой правкой.
* Если задача понятна и не попадает в критичные исключения, сразу выполняй минимальную проверку состояния проекта и работай автономно.
* Короткий план нужен только если задача неоднозначная, потенциально рискованная или затрагивает критичные области.

* Не меняй цены, контакты, юридический смысл текста без подтверждения.

* Если задача касается архитектурного refactor, можно менять общие шаблоны и связанные route/data-файлы, если это прямо относится к задаче.
* После правки напиши: изменённые файлы, что изменено, какие проверки прошли, commit/push.

## Системные подтверждения Antigravity / IDE

Если Antigravity или IDE показывает системное подтверждение команды, это не считается вопросом Claude к пользователю. Это отдельная защита среды.

Claude должен уменьшать количество таких запросов:

* группировать безопасные read-only команды;
* не запускать лишние команды;

## Правила команд для Antigravity

Claude должен запускать terminal commands так, чтобы они совпадали с Allow List Antigravity и не вызывали лишние системные подтверждения.

Проект уже открыт в корневой папке workspace. Поэтому Claude не должен без необходимости добавлять абсолютный переход:

`cd C:\Users\Andrii\omobonus-serwis; ...`

Вместо этого запускай команды напрямую из текущей директории:

`npm run build`

`npx tsc --noEmit`

`git status --short`

`git diff --stat`

`git branch --show-current`

Если нужно выполнить несколько проверок, предпочитай простые отдельные команды, которые есть в Allow List, вместо длинных compound-команд с абсолютными путями.

Не используй абсолютные `cd C:\...` команды, если терминал уже находится в корне проекта.

Не используй сложные PowerShell-команды с `cd`, pipes и path operations без необходимости, потому что Antigravity может запросить ручное подтверждение.

Для проверки TypeScript используй:

`npx tsc --noEmit`

Для build используй:

`npm run build`

Для Lighthouse используй:

`npx lighthouse`

Для PageSpeed API используй разрешённые network/fetch команды, если они уже разрешены Antigravity.

Если Antigravity всё равно требует системное подтверждение безопасной команды, это не считается вопросом Claude. После разрешения среды Claude должен продолжить выполнение и не менять план задачи.

## Дополнительные правила для уменьшения подтверждений Antigravity

Claude должен предпочитать простые команды, которые совпадают с Allow List Antigravity.

Не используй без необходимости PowerShell pipes и сложные compound-команды:

`2>&1 | Select-Object -Last 30`

`cd C:\...\project; command`

`command | Select-Object`

`command; command; command`

Если нужна проверка, запускай простую команду напрямую:

`npm run build`

`npx tsc --noEmit`

`git status --short`

`git diff --stat`

`git branch --show-current`

Если нужно увидеть последние строки вывода, сначала запусти обычную команду. Не добавляй `Select-Object -Last`, если без этого можно обойтись.

Не используй `/lint` как Claude skill. Если нужно проверить lint:

1. прочитай `package.json`;
2. найди реальный script;
3. запускай только существующий npm-script, например `npm run lint`, если он реально есть.

Для PageSpeed/performance-задач:

* не считай localhost Lighthouse финальным результатом;
* localhost можно использовать только для предварительной проверки;
* финальный критерий — PageSpeed/Lighthouse на preview или production URL;
* если production PageSpeed почти не улучшился, продолжай искать реальный LCP bottleneck;
* не объявляй задачу завершённой только по localhost-метрикам.

Если Antigravity всё равно требует подтверждение безопасной команды, Claude должен после разрешения продолжить задачу без изменения плана.
