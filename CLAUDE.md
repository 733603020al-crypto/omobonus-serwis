# CLAUDE.md

## Правила работы

Отвечай на русском. Работай экономно по токенам.

Работай профессионально и критично. Если данных недостаточно — скажи об этом или задай короткий уточняющий вопрос. Не делай вид, что уверен, если это не так.

## Экономия токенов

* Не анализируй весь проект без прямой просьбы.
* Не читай все файлы подряд.
* Не читай отчёты, логи и старые markdown-файлы без запроса.
* Для каждой задачи сначала найди минимальный набор нужных файлов.
* Если задача обычная и нужно открыть больше 3 файлов — сначала спроси разрешение.
* Если задача касается языков, SEO, routes, canonical, hreflang или архитектурного refactor, можно открыть больше 3 файлов без вопроса, но только релевантные файлы.
* Если задача является audit/check/inspection/read-only анализом, используй правила из раздела `Read-only audit mode`.
* Не показывай полный код файлов. Показывай только краткий diff или изменённые фрагменты.
* Отвечай коротко: что нашёл, что предлагаешь, какие файлы тронуть.

## Read-only audit mode

Этот режим применяется, если задача сформулирована как:

* audit;
* проверка лишних файлов;
* поиск мусора;
* поиск неиспользуемых файлов;
* inspection;
* read-only анализ;
* проверка структуры проекта без изменений.

В этом режиме работай автономно до финального отчёта.

Разрешено без дополнительного вопроса выполнять read-only проверки:

* `git status --short`;
* `git branch --show-current`;
* `git log --oneline`;
* `git diff --name-only`;
* `git diff --stat`;
* `ls`;
* `find`;
* `rg`;
* `du`;
* `wc`;
* `head`;
* `tail`;
* чтение обычных файлов проекта, если они нужны для проверки использования;
* проверка импортов, ссылок, путей, ассетов, компонентов, data/config файлов.

В этом режиме можно открыть больше 3 релевантных файлов без вопроса, если это нужно для audit-задачи.

Не спрашивай промежуточные подтверждения для read-only команд. Группируй проверки разумно и продолжай до итогового отчёта.

Спрашивай подтверждение только если нужно:

* удалить файл или папку;
* изменить файл;
* сделать commit;
* сделать push;
* установить пакет;
* выполнить `git reset`;
* выполнить `git rebase`;
* выполнить `git clean`;
* выполнить команду, которая может изменить проект;
* читать `.env*`;
* менять код, SEO, URL, canonical, hreflang, юридические тексты, цены, контакты или форму отправки.

Важно:

* Ничего не удаляй без отдельного подтверждения.
* Ничего не меняй без отдельного подтверждения, если задача явно read-only.
* Не делай commit/push без явного разрешения в текущем задании.
* Не предлагай удалять `node_modules` как часть очистки проекта.
* Не предлагай удалять `.next` как изменение проекта.
* Не предлагай удалять `public/images` без доказательства, что конкретный файл нигде не используется.
* Не предлагай удалять `CLAUDE.md`.
* Не предлагай удалять файлы правил Claude.
* Не читай `.env*`.

## Безопасность

* Не читай `.env`, `.env.local`, `.env.production`, `.env.development`, ключи, токены, пароли и секреты без моего разрешения.
* Не выводи секреты в ответ.
* Не устанавливай пакеты без подтверждения.
* Не запускай опасные команды: удаление файлов, очистка базы, `git reset`, `git rebase`, `git clean`, деплой, push в `master`, отправка секретов наружу.
* `git push origin test` разрешён только если это явно указано в текущем задании и правила проекта это позволяют.
* `master` нельзя трогать вообще.
* Работай только в ветке `test`, если в текущем задании не указано иное.
* Если текущая ветка не `test`, сначала сообщи об этом и не продолжай без подтверждения.

## Изменение сайта

* Перед правкой дай короткий план, если текущая задача не указана как автономная.
* Если задача явно указана как автономная, сначала сделай минимальную проверку состояния проекта, затем выполняй задачу без промежуточных вопросов.
* Меняй минимальное количество файлов.
* Не делай рефакторинг “заодно”.
* Не меняй цены, контакты, юридические страницы без подтверждения.
* Тексты и SEO можно менять только если текущая задача явно касается текстов, локализации, SEO или синхронизации языковых версий.
* После правки напиши: изменённые файлы, что изменено, как проверить.
