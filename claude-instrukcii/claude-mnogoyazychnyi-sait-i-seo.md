# Правила многоязычного сайта и SEO

Этот файл нужно читать перед любой задачей, связанной с языками, переводами, SEO, sitemap, canonical, hreflang, routes или добавлением новых языков.

Перед любыми изменениями молча проверь текущую git-ветку.

Работать можно только в ветке `test`.

Если текущая ветка не `test`, остановись и ничего не меняй.

Если текущая ветка `test`, работай дальше автономно и не спрашивай про ветку каждый раз.

Ветка `master` является опубликованной версией сайта и подключена к Vercel. Ветку `master` нельзя трогать вообще.

---

## 0. Автономная работа при архитектурном refactor

При задачах архитектурного refactor Claude должен работать автономно и не задавать промежуточные вопросы, если действие безопасно и соответствует правилам проекта.

Разрешено самостоятельно:

* читать нужные файлы;
* выносить повторяющийся JSX в общий шаблон;
* подключать PL/UK route-файлы к общему шаблону;
* запускать `npx tsc --noEmit`;
* запускать `npm run build`;
* показывать `git diff --name-only`;
* показывать `git diff --stat`.

Commit/push разрешён только если это явно указано в текущем задании. Если commit/push разрешён, то только в ветку `test`.

Перед `git add`, `git commit` или `git push` Claude должен проверить:

* текущая ветка — `test`;
* `git status --short`;
* список изменённых файлов соответствует текущей задаче;
* среди изменённых файлов нет неожиданных файлов;
* нет изменений в `master`;
* нет изменений в ценах, контактах, юридических текстах, URL, sitemap, robots.txt или отправке формы, если это не было явно частью задачи.

Если текущая задача явно разрешает commit/push и изменены только ожидаемые файлы, Claude может самостоятельно выполнить:

* `git add`;
* `git commit`;
* `git push origin test`.

Если в `git status --short` есть неожиданные файлы, Claude должен остановиться и показать список файлов.

Нельзя использовать `git add -A`, если Claude не проверил список изменённых файлов.


Остановиться и спросить нужно только если:

* текущая ветка не `test`;
* требуется менять `master`;
* требуется менять URL, slug, canonical или hreflang;
* требуется менять цены;
* требуется менять контакты;
* требуется менять юридические тексты;
* требуется менять отправку формы;
* требуется удалить существующий блок;
* build/typecheck не проходит и причина неочевидна;
* для выполнения задачи нужно изменить смысл SEO-текстов;
* изменение выходит за рамки текущего задания.

Во всех остальных случаях Claude должен сам выбрать безопасный вариант, выполнить задачу, проверить результат и дать короткий отчёт.


## 1. Главный принцип сайта

Для чего этот блок:
чтобы всегда было понятно, какая версия сайта является основной.

Польская версия сайта является основной и исходной.

Корневые маршруты относятся к польской версии:

* `/`
* `/uslugi/...`
* `/kontakt`
* `/regulamin`
* `/polityka-prywatnosci`

Остальные языки являются локализациями польской структуры:

* украинский: `/uk`, если украинская версия реально существует;
* русский: `/ru`, если русская версия реально существует;
* английский: `/en`, если английская версия реально существует.

Нельзя считать язык существующим только потому, что он указан в правилах. Язык считается существующим только если в проекте реально есть соответствующие маршруты, страницы и данные.

---

## 2. Архитектура многоязычности

Для чего этот блок:
чтобы не создавать отдельные копии сайта под каждый язык.

Нельзя создавать независимые копии дизайна, компонентов, layout и бизнес-логики для каждого языка, если можно использовать общую структуру, общий компонент или языковые data-файлы.

Правильный подход:

* один общий дизайн;
* общие компоненты там, где это возможно;
* общая логика;
* общая структура страниц;
* разные языковые тексты;
* разные SEO-данные;
* разные языковые URL;
* синхронизация языковых версий по структуре, если польская версия была изменена.

Плохой подход:

* копировать польскую страницу целиком как независимую верстку и дальше поддерживать её отдельно;
* создавать отдельную верстку для каждого языка без необходимости;
* менять дизайн только в одной языковой версии;
* делать разные структуры страниц без согласования;
* оставлять украинскую, русскую или английскую версию визуально устаревшей относительно польской версии.

---

### 2.1. Архитектура общих шаблонов для языковых версий

Для многоязычных страниц нельзя создавать независимые копии JSX-разметки для каждого языка, если страницу можно построить через общий шаблон.

Правильная архитектура:

* общий шаблон страницы;
* отдельные данные для каждого языка;
* отдельное SEO для каждого языка;
* отдельные URL/canonical/hreflang для каждого языка.

Пример правильной схемы:

* `HomePageTemplate` — общий шаблон главной страницы;
* `ServicePageTemplate` — общий шаблон страницы услуги;
* PL route-файл передаёт польские данные;
* UK route-файл передаёт украинские данные;
* будущие RU/EN route-файлы должны использовать тот же общий шаблон.

Route-файлы должны быть тонкими. Они могут содержать:

* `metadata`;
* `generateMetadata`;
* `generateStaticParams`;
* импорт языковых данных;
* передачу props в общий шаблон.

Route-файлы не должны содержать большую повторяющуюся JSX-разметку страницы, если такая разметка уже есть или может быть вынесена в общий шаблон.

Если нужно изменить структуру страницы, порядок секций, дизайн блока, CTA, hero, services, accordion, footer-зону или общую логику отображения, изменение нужно делать в общем шаблоне, а не только в польском или украинском `page.tsx`.

Языковые отличия должны передаваться через props/data/i18n:

* `locale`;
* `basePath`;
* `href`;
* `labels`;
* `headings`;
* `seoBlocks`;
* `servicesData`;
* `footerT`;
* `heroT`;
* `aboutT`;
* `imageAlt`;
* `metadata`;
* `canonical`;
* `hreflang`;
* optional language-specific section.

Если у одного языка есть допустимое исключение, например блок `ContactUk` только на украинской главной странице, нельзя из-за этого дублировать всю страницу. Такое исключение нужно передавать в общий шаблон через optional prop/slot, например `contactSection`, `afterCta` или аналогичный механизм.

Запрещено:

* менять только польскую структуру и оставлять украинскую копию старой;
* создавать новый отдельный JSX-шаблон для UK/RU/EN, если уже есть общий шаблон;
* переносить SEO-тексты в общий hardcoded компонент;
* смешивать PL/UK/RU тексты внутри общего шаблона;
* удалять языковые исключения без явного задания;
* менять URL, slug, canonical или hreflang без отдельного разрешения.

Разрешено:

* держать SEO-тексты отдельно для каждого языка;
* адаптировать SEO-текст под язык;
* добавлять поисковые фразы отдельно для каждой языковой версии;
* использовать общий layout и разные языковые данные;
* оставлять специальные языковые блоки как optional props, если они нужны только одному языку.

Главное правило:

скелет страницы общий, содержимое и SEO — отдельные для каждого языка.


## 3. Что может отличаться между языками

Для чего этот блок:
чтобы понимать, что переводится, а что должно оставаться общим.

Между языковыми версиями могут отличаться:

- тексты на странице;
- тексты кнопок;
- title;
- description;
- H1;
- H2;
- FAQ;
- alt-тексты изображений, если они языковые;
- SEO-тексты;
- metadata;
- canonical URL;
- hreflang URL;
- языковые data-файлы;
- slug, если это отдельно согласовано.

Между языковыми версиями не должны отличаться без согласования:

- дизайн;
- сетка;
- порядок основных блоков;
- структура компонентов;
- логика формы;
- логика отправки заявки;
- базовые UI-компоненты.

### 3.1. Видимые тексты в общих компонентах

Для чего этот блок:
чтобы в украинской, русской или английской версии не оставались польские тексты из общих компонентов.

Если общий компонент используется в нескольких языковых версиях и содержит видимый текст, этот текст нельзя оставлять захардкоженным только на одном языке.

Примеры таких текстов:

- подписи в hero-блоке;
- подписи статистики;
- бейджи Google Rating;
- тексты карточек;
- тексты кнопок;
- подписи CTA;
- короткие UI-лейблы;
- подписи преимуществ;
- тексты внутри общих секций.

Нельзя просто заменить польский текст внутри общего компонента на украинский, русский или английский, потому что это сломает польскую версию.

Правильный подход:

- вынести текст в props;
- или вынести текст в языковой data/i18n-файл;
- или передавать текст из конкретной языковой страницы;
- сохранить общий дизайн и структуру компонента;
- сохранить польский текст для польской версии;
- передать украинский текст для украинской версии;
- предусмотреть возможность русской и английской версии.

Если задача касается локализации, Claude может менять общий компонент только технически, чтобы сделать его языковым.

При таком изменении нельзя менять:

- внешний вид компонента;
- layout;
- отступы;
- цвета;
- шрифты;
- структуру DOM без необходимости;
- польские тексты польской версии;
- бизнес-логику компонента.

Если общий компонент содержит польский текст и используется на `/uk`, `/ru` или `/en`, Claude должен не оставлять этот текст как есть, а сделать компонент языковым безопасным способом.

---

### 3.2. Точность перевода

Для чего этот блок:
чтобы Claude не переписывал тексты свободно и не заменял перевод новым маркетинговым или SEO-текстом.

Если задача касается перевода или синхронизации языковой версии, перевод должен быть близким к оригиналу.

По умолчанию Claude должен делать не свободный рерайтинг, а точный перевод с сохранением смысла, структуры и тона оригинального польского текста.

Нельзя без прямого разрешения пользователя:

- заменять философский текст рекламным текстом;
- заменять текст "О нас" SEO-текстом;
- добавлять услуги, которых нет в оригинальном фрагменте;
- добавлять город, если его нет в оригинальном фрагменте;
- добавлять обещания качества, если их нет в оригинальном фрагменте;
- убирать религиозные, исторические или смысловые образы, если они есть в оригинале;
- упрощать текст так, что меняется смысл;
- писать "примерно о том же", если пользователь просит перевод.

Если дословный перевод звучит неестественно, можно сделать его естественным для целевого языка, но нельзя менять смысл.

Правильный подход:

- сохранить основную мысль каждого предложения;
- сохранить порядок смыслов;
- сохранить тон оригинала;
- не добавлять новые факты;
- не удалять важные образы;
- не превращать перевод в SEO-рерайтинг;
- не заменять брендовый текст новым текстом.

Если польский оригинал говорит:

"Jesteśmy zespołem, który wierzy, że praca może być również pomocą i służbą innym ludziom."

то украинский перевод должен передавать именно эту мысль:

"Ми — команда, яка вірить, що праця може бути також допомогою і служінням іншим людям."

А не заменяться на новый текст про сервис компьютеров, ноутбуков, принтеров или город.

Перед завершением перевода Claude должен сравнить польский оригинал и целевой перевод по смыслу.

Если целевой текст содержит факты, которых нет в польском оригинале, Claude должен исправить перевод или отдельно сообщить, что это уже не перевод, а адаптация.

## 4. Правило изменения польской версии

Для чего этот блок:
чтобы языковые версии не устаревали относительно польской версии.

Польская версия является не только источником при создании новых языковых страниц.

Польская версия является постоянным источником правды для всех существующих языковых версий.

Это значит:

если в польской версии меняется текст, структура, layout, блок, секция, кнопка, CTA, форма, порядок блоков, карточка, заголовок, подзаголовок, SEO-текст или компонент, Claude должен проверить соответствующие языковые версии и синхронизировать их с актуальной польской версией.

Нельзя обновлять только польскую версию и оставлять украинскую, русскую или английскую версию в старом состоянии, если соответствующая страница реально существует.

---

### 4.1. Что считается изменением польской версии

Изменением польской версии считается:

* изменение текста;
* изменение H1;
* изменение H2;
* изменение subtitle;
* изменение CTA;
* изменение кнопки;
* изменение формы;
* изменение порядка секций;
* добавление новой секции;
* удаление секции;
* замена одного блока другим;
* изменение карточек;
* изменение списка услуг;
* изменение FAQ;
* изменение SEO-текста;
* изменение metadata;
* изменение title;
* изменение description;
* изменение layout;
* изменение компонента;
* изменение логики отображения блока;
* изменение ссылки;
* изменение текста в общем компоненте.

Если такое изменение сделано в польской версии, соответствующие языковые версии должны быть проверены и обновлены.

---

### 4.2. Как синхронизировать языковые версии

При синхронизации Claude должен сравнивать не старый украинский текст с новым украинским текстом, а актуальную польскую версию с целевой языковой версией.

Правильная схема:

* текущая польская версия → украинская версия;
* текущая польская версия → русская версия;
* текущая польская версия → английская версия.

Неправильная схема:

* старая украинская версия → новая украинская версия;
* украинская версия → русская версия;
* старая версия файла → новая версия файла без проверки польского источника;
* переводить только найденные строки, не сравнивая актуальную польскую страницу.

Claude должен всегда брать за основу реальные актуальные файлы польской версии в текущей ветке `test`.

Скриншоты могут быть подсказкой, но главный источник правды — реальные файлы проекта.

---

### 4.3. Если изменилась польская главная страница

Если изменилась польская главная страница `/`, Claude должен проверить и при необходимости обновить:

* `/uk`, если украинская версия реально существует;
* `/ru`, если русская версия реально существует;
* `/en`, если английская версия реально существует.

Проверять нужно не только перевод, но и:

* структуру страницы;
* порядок блоков;
* CTA-блоки;
* форму;
* hero;
* about;
* services;
* FAQ;
* contact;
* footer CTA;
* карточки;
* кнопки;
* маленькие section labels;
* uppercase labels;
* тексты в общих компонентах.

---

### 4.4. Если изменилась польская страница услуги

Если изменилась польская страница услуги `/uslugi/...`, Claude должен проверить соответствующую языковую страницу:

* `/uk/uslugi/...`, если она реально существует;
* `/ru/uslugi/...`, если она реально существует;
* `/en/uslugi/...`, если она реально существует.

Проверять нужно:

* title;
* subtitle;
* H1;
* description;
* SEO-текст;
* FAQ;
* цены;
* карточки;
* accordion;
* CTA;
* fallback-тексты;
* section labels;
* тексты в `services-data`;
* тексты в языковых data-файлах;
* тексты в общих компонентах.

Если польская страница услуги изменилась, языковая версия не должна оставаться переводом старой польской версии.

---

### 4.5. Тексты нужно не просто переводить, а синхронизировать

Если польский текст был изменён, Claude должен обновить соответствующий текст в языковых версиях.

Нельзя оставлять старый украинский, русский или английский текст только потому, что он уже был переведён раньше.

Если польский оригинал изменился, старый перевод считается устаревшим.

Claude должен:

* найти актуальный польский текст;
* найти соответствующий текст в целевой языковой версии;
* проверить, соответствует ли он актуальному польскому тексту;
* если не соответствует, обновить перевод;
* сохранить смысл польского оригинала;
* не добавлять новые факты;
* не делать свободный SEO-рерайтинг вместо перевода.

---

### 4.6. Структурные изменения тоже нужно переносить

Синхронизация языковых версий касается не только текста.

Если в польской версии изменился блок, структура или layout, соответствующая языковая версия тоже должна быть обновлена.

Пример:

Если в польской версии удалили блок, соответствующая языковая версия тоже должна быть проверена на удаление или замену этого блока.

Если изменение затрагивает бизнес-логику, форму, отправку заявки, цены, контакты или юридический текст, Claude должен действовать осторожно и не придумывать решение самостоятельно.

Но если пользователь уже утвердил изменение в польской версии и задача состоит в синхронизации языков, соответствующая языковая версия должна быть приведена к той же актуальной структуре.

### 4.10. Как изменять польскую версию после перехода на общие шаблоны

Если изменение касается общего шаблона, оно должно быть проверено минимум на PL и UK.


## 5. Canonical

Для чего этот блок:
чтобы Google понимал основной адрес каждой конкретной языковой страницы.

Каждая реальная языковая страница должна иметь self-canonical.

Пример:

* `/uslugi/serwis-laptopow` canonical указывает на `/uslugi/serwis-laptopow`;
* `/uk/uslugi/serwis-laptopow` canonical указывает на `/uk/uslugi/serwis-laptopow`;
* `/ru/uslugi/serwis-laptopow` canonical указывает на `/ru/uslugi/serwis-laptopow`;
* `/en/uslugi/serwis-laptopow` canonical указывает на `/en/uslugi/serwis-laptopow`.

Нельзя делать так, чтобы все языковые версии указывали canonical только на польскую страницу.

---

## 6. Hreflang

Для чего этот блок:
чтобы Google понимал, что разные URL являются версиями одной страницы на разных языках.

Каждая реальная языковая страница должна иметь hreflang на все существующие языковые версии этой же страницы.

Для главной страницы:

* `pl` ведет на `/`;
* `uk` ведет на `/uk`;
* `ru` ведет на `/ru`, только если страница реально существует;
* `en` ведет на `/en`, только если страница реально существует;
* `x-default` ведет на польскую версию `/`.

Нельзя добавлять hreflang на URL, которых нет.

---

## 7. HTML lang

Для чего этот блок:
чтобы браузер и поисковики понимали язык страницы.

Каждая языковая версия должна иметь правильный язык HTML:

* польская версия: `pl`;
* украинская версия: `uk`;
* русская версия: `ru`;
* английская версия: `en`.

## 8. Sitemap

Для чего этот блок:
чтобы поисковики видели все реальные страницы сайта.

В sitemap должны попадать только реальные существующие страницы.

Если страницы `/ru/...` или `/en/...` еще не созданы, их нельзя добавлять в sitemap.

При добавлении нового языка нужно проверить:

* главную страницу языка;
* страницы услуг;
* контакт;
* юридические страницы, если они реально существуют на этом языке;
* canonical;
* hreflang.

## 9. Индексация и HTML

Для чего этот блок:
чтобы основной SEO-контент был виден Google сразу.

Google должен видеть основной текст страницы сразу в HTML.

В HTML должны быть доступны:

Например, Google Reviews может грузиться динамически, если основные SEO-тексты страницы уже есть в HTML.

* `ContactUk` является допустимым language-specific блоком только для `/uk` и должен передаваться через optional prop/slot, а не через дублирование всей страницы.

Нельзя возвращать главную страницу или страницы услуг к схеме независимых копий JSX для PL и UK.

Если нужно изменить общую структуру главной страницы — менять `HomePageTemplate`.

Если нужно изменить общую структуру страницы услуги — менять `ServicePageTemplate`.

Если нужно изменить CTA главной страницы — менять общий `HomeCta` и передавать языковые тексты через props/data.

Если нужно изменить только польский, украинский, русский или английский текст — менять соответствующие языковые данные, а не общий шаблон.
