# Правила дизайна и работы с проектом Omobonus

## Главная идея

Проект Omobonus уже имеет готовый дизайн и рабочую структуру.

Задача Claude Designer — аккуратно дорабатывать существующий сайт, а не создавать новый дизайн с нуля.

Главный источник правды — реальные файлы проекта, а не скриншоты.

---

## Ветки GitHub и безопасность сайта

В проекте есть две основные ветки:

- `master` — настоящий сайт. Эта ветка подключена к Vercel и публикуется в интернет.
- `test` — черновик для работы Claude Designer.

Claude Designer работает только в ветке `test`.

Claude Designer не должен самостоятельно:
- менять ветки;
- переключаться на `master`;
- делать pull;
- делать push;
- делать merge;
- создавать Pull Request;
- переносить изменения между ветками.

Я сам управляю ветками GitHub вручную.

Перед началом работы Claude Designer должен проверить и написать мне:
- в какой ветке он сейчас находится;
- что это именно ветка `test`;
- какие файлы он собирается менять.

Если Claude Designer видит, что он находится в ветке `master`, он должен остановиться и ничего не менять.

---

## Работа двух Claude

В проекте могут работать два помощника:

- Claude в Antigravity — работает с настоящим проектом под моим контролем.
- Claude Designer — работает только в ветке `test`.

Нельзя двум помощникам одновременно менять одну и ту же страницу или одни и те же файлы.

Если Claude Designer хочет изменить страницу, которую сейчас редактируют в Antigravity, он должен остановиться и спросить меня.

Перед изменениями Claude Designer должен написать:
- какую страницу или секцию будет менять;
- какие файлы будет менять;
- может ли это пересечься с работой в Antigravity.

---

## Доработка существующего дизайна

Этот проект уже имеет готовый стиль.

Задача — дорабатывать существующий сайт, а не создавать новый дизайн.

Запрещено без моего разрешения:
- менять общий стиль сайта;
- менять цвета;
- менять шрифты;
- менять структуру страниц;
- создавать новые кнопки, карточки и секции, если похожие уже есть;
- заменять существующие компоненты новыми;
- делать редизайн страницы целиком;
- менять header;
- менять footer;
- менять меню;
- менять глобальные стили;
- менять конфиги проекта.

Если нужно что-то улучшить, сначала предложи вариант, но не внедряй его сразу.

Перед внедрением покажи:
- визуальное представление;
- что именно изменится;
- почему это будет лучше;
- какие файлы будут затронуты.

Жди моего подтверждения.

---

## Сохранение стиля сайта

При любых изменениях нужно сохранять текущий стиль Omobonus:

- тёмный фон;
- старинная фактура;
- золотые акценты;
- serif-шрифты;
- спокойная премиальная подача;
- существующая логика карточек, кнопок и секций;
- общий визуальный характер сайта.

Нельзя делать сайт:
- современным “с нуля”;
- минималистичным;
- ярким;
- корпоративным;
- похожим на шаблон;
- похожим на другой сайт;

если я этого прямо не попросил.

---

## Использование готовых компонентов

Сначала ищи похожий готовый компонент в проекте.

Если готовый компонент есть — используй его.

Если готового компонента нет — сначала спроси разрешение на создание нового.

Не создавай новые компоненты, если можно использовать существующие.

Не создавай новые варианты кнопок, карточек, секций и блоков, если в проекте уже есть похожие.

---

## Визуальные предложения до кода

Если я спрашиваю:

- “как лучше”;
- “как красивее”;
- “что посоветуешь”;
- “как это будет выглядеть”;
- “покажи варианты”;

сначала предложи визуальный вариант, а не меняй код.

Визуальный вариант может быть:
- простым текстовым макетом;
- описанием расположения элементов;
- изображением или preview, если это возможно.

Код меняй только после моего явного подтверждения:

- “да, внедряй”;
- “применяй”;
- “можно менять”;
- “вноси в код”.

---

## Работа только с нужной страницей

Если задача касается одной страницы, меняй только эту страницу.

Не меняй другие страницы без прямого разрешения.

Если нужная правка затрагивает общий компонент, сначала предупреди меня:

- какой компонент;
- где он используется;
- какие страницы могут измениться.

Без моего подтверждения нельзя менять общий компонент.

---

## Не работать по догадкам

Скриншоты — только подсказка.

Главный источник правды — реальные файлы проекта.

Если скриншот и код отличаются, сначала спроси меня, что считать правильным.

Если не уверен, какой файл отвечает за нужный блок, сначала найди файл и покажи мне путь.

Не меняй похожий блок наугад.

Если видишь несколько похожих компонентов, сначала спроси меня, какой именно нужно менять.

---

## Перед изменением файлов

Перед изменением файлов Claude Designer должен сначала написать коротко:

1. Что он понял из задачи.
2. Какую страницу или секцию будет менять.
3. Какие файлы планирует изменить.
4. Какие элементы нельзя трогать.
5. Может ли изменение повлиять на другие страницы.
6. Нужна ли моя команда “можно менять”.

Без моего подтверждения нельзя начинать правку.

---

## Когда нужно остановиться

Остановись и спроси меня, если:

- нужно менять общий компонент;
- нужно менять глобальные стили;
- нужно менять несколько страниц;
- нужно менять header, footer или меню;
- нужно менять цвета или шрифты сайта;
- нужно добавить новую библиотеку;
- нужно изменить конфиги проекта;
- есть риск, что изменение повлияет на настоящий сайт;
- ты не уверен, какой файл отвечает за нужный блок;
- задача может конфликтовать с работой Claude в Antigravity.

---

## После изменения файлов

После каждой правки Claude Designer должен написать:

- какие файлы были изменены;
- что именно изменилось;
- что мне проверить глазами в preview;
- есть ли риск влияния на другие страницы.

Не показывай полный код файлов без моей просьбы.

Показывай только краткое описание, изменённые фрагменты или diff.

---

## Тексты и польский язык

Не переписывай польские тексты свободно.

Если нужно изменить текст, сначала предложи новый вариант и дождись моего подтверждения.

Не меняй:
- цены;
- контакты;
- юридические тексты;
- SEO-тексты;
- названия услуг;
- условия скидок;

без моего прямого разрешения.

---

## Экономия и порядок работы

Работай точечно.

Не анализируй весь проект без необходимости.

Не читай все файлы подряд.

Сначала найди минимальный набор файлов, нужных для задачи.

Если нужно открыть больше 3 файлов — сначала спроси разрешение.

Не делай рефакторинг “заодно”.

Не улучшай то, о чём я не просил.

---

## Главное правило в конце

Claude Designer не должен удивлять меня изменениями.

Сначала объясни.
Потом покажи вариант.
Потом дождись разрешения.
Только потом меняй код.
